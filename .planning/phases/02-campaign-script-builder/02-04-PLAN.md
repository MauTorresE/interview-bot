**2. `src/app/interview/[token]/actions.ts`** — Server action for consent recording.

Mark `'use server'`. This uses admin client since respondents are NOT authenticated users (Pitfall 2 from RESEARCH.md).

**`recordConsent(token: string): Promise<{ success?: boolean; error?: string }>`**
- Create admin client (service role key).
- Look up respondent by invite_token: `admin.from('respondents').select('id, status, campaign_id, campaigns(id, status)').eq('invite_token', token).single()`.
- If not found: return `{ error: 'invalid' }`.
- If respondent.status !== 'invited': return `{ error: 'already_used' }` (prevent re-consent).
- If campaign status === 'archived': return `{ error: 'campaign_archived' }`.
- Update respondent: `admin.from('respondents').update({ status: 'in_progress', consent_given_at: new Date().toISOString() }).eq('id', respondent.id)`.
- Return `{ success: true }`.

Also handle reusable campaign tokens:
**`validateToken(token: string): Promise<{ valid: boolean; type: 'respondent' | 'campaign' | null; campaignName: string | null; error?: string }>`**
- First check respondents table for invite_token match.
- If found: return `{ valid: true, type: 'respondent', campaignName: campaign.name }`.
- If not found: check campaigns table for reusable_invite_token match where reusable_invite_enabled = true.
- If found: return `{ valid: true, type: 'campaign', campaignName: campaign.name }`.
- If neither: return `{ valid: false, type: null, campaignName: null, error: 'invalid' }`.

**`recordConsentForReusableLink(token: string, name: string): Promise<{ success?: boolean; error?: string }>`**
- Admin client lookup campaign by reusable_invite_token where reusable_invite_enabled = true.
- If not found: return error.
- Create new respondent: `admin.from('respondents').insert({ campaign_id, org_id: campaign.org_id, name, email: null, status: 'in_progress', consent_given_at: new Date().toISOString() })`.
- Return `{ success: true }`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public internet -> Consent page | Unauthenticated respondent accesses via invite token |
| Consent page -> Database | Admin client writes consent_given_at to respondent record |
| Client -> Config server action | Campaign settings updated by authenticated user |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-11 | Information Disclosure | Consent page token lookup | mitigate | Only returns campaign name (not org data, not other respondents). Admin client used server-side only. Token validated before any data returned |
| T-02-12 | Tampering | recordConsent action | mitigate | Checks respondent.status === 'invited' before allowing consent; prevents re-consent or status manipulation |
| T-02-13 | Denial of Service | Reusable link respondent creation | mitigate | Each call creates one respondent; rate limiting at Next.js/Vercel level. Reusable link can be disabled by org admin via toggle |
| T-02-14 | Information Disclosure | Invite token enumeration | accept | UUIDs are 128-bit random (gen_random_uuid()). Brute-force infeasible. Non-matching tokens return generic "Enlace no valido" |
| T-02-15 | Elevation of Privilege | updateCampaignConfig | mitigate | Auth check + org_id from JWT; admin client update includes org_id condition |
</threat_model>
