    **Invite Server Action (`src/app/(dashboard)/settings/invite/actions.ts`):**
    - `'use server'` directive
    - Validate email with `inviteSchema`
    - Check user is org member (get org_id from user's app_metadata)
    - Insert into `org_invites`: `{ org_id, email, invited_by: user.id }`
    - The token and expires_at are auto-generated by database defaults
    - For v1: the invite "link" is constructed as `{origin}/api/invites/accept?token={token}`
    - NOTE: Actual email sending is deferred — for now, return the invite link in the success response so it can be shared manually. Log it or show it in a toast.
    - Return `{ success: true, inviteLink: string }` or `{ error: string }`

    **Invite Accept Endpoint (`src/app/api/invites/accept/route.ts`):**
    - GET handler that accepts `?token=xxx` query param
    - Look up invite in `org_invites` where `token = xxx AND accepted_at IS NULL AND expires_at > now()`
    - If not found or expired: redirect to `/auth/login` with error param
    - If found:
      - Get authenticated user from Supabase (the invitee must be logged in)
      - If not authenticated: redirect to `/auth/signup?invite={token}` (they can sign up first)
      - If authenticated:
        1. Insert into `org_members`: `{ org_id: invite.org_id, user_id: user.id, role: 'member' }`
        2. Update `org_invites`: set `accepted_at = now()`
        3. Update user's `app_metadata.org_id` to the new org via admin client
        4. Redirect to `/campaigns`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Landing page -> Auth | Public page links to auth flow |
| Invite link -> Accept endpoint | Token-based access to join org |
| Settings actions -> Supabase Admin | Org creation and invite use service_role |
| Org switcher -> Server action | Org switch updates JWT app_metadata |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege | Invite acceptance | mitigate | Token validated: must exist, not expired (48h), not already accepted. User must be authenticated. |
| T-03-02 | Information Disclosure | Invite tokens in URL | accept | Tokens are UUIDs (not guessable), expire in 48h, single-use. Acceptable for v1 invite flow. |
| T-03-03 | Tampering | Org creation | mitigate | Uses admin client in server action only. User gets 'owner' role automatically. app_metadata set server-side. |
| T-03-04 | Spoofing | Org switching | mitigate | switchOrg verifies user is member of target org before updating app_metadata. Uses admin client server-side. |
| T-03-05 | Denial of Service | Invite spam | accept | No rate limiting on invites in v1. Acceptable for closed beta. Monitor usage. |
| T-03-06 | Information Disclosure | Landing page | accept | Public page with no sensitive data. Only shows marketing copy. |
</threat_model>
