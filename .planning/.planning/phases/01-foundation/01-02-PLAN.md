<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Auth form -> Server Action | User-supplied credentials cross to server |
| Signup action -> Supabase Admin | Org creation uses service_role key |
| Browser -> Dashboard routes | Must be authenticated to access |
| Password reset email -> Callback | Token exchange for session |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | Login form | mitigate | Validate with zod schema server-side; rate limiting handled by Supabase Auth |
| T-02-02 | Information Disclosure | Error messages | mitigate | Generic error messages ("Credenciales incorrectas") — never reveal whether email exists |
| T-02-03 | Elevation of Privilege | Signup auto-org creation | mitigate | Admin client (service_role) only in server action with 'use server' directive; never in client code |
| T-02-04 | Tampering | Password reset flow | mitigate | Use Supabase's built-in PKCE flow via exchangeCodeForSession; validate code server-side |
| T-02-05 | Repudiation | Auth state | accept | Supabase Auth logs sign-in events; sufficient for v1 |
| T-02-06 | Spoofing | Dashboard access | mitigate | Dashboard layout fetches user via getUser() (server-side, secure); redirects if null |
</threat_model>
